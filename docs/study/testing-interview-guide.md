## 一、后端面试里测试考什么

面试官看重的不是“会不会跑 `pytest`”，而是：

1. **质量意识**：是否有“改代码要有验证”的习惯
2. **测试分层意识**：知道单测、集成、端到端各测什么
3. **可维护性意识**：测试是否稳定、可读、可复用
4. **风险意识**：核心链路是否有回归保护

## 二、测试金字塔与测试类型

## 2.1 测试金字塔

```
        E2E（少）
      Integration（中）
    Unit Tests（多）
```

原则：
- 单元测试数量最多，执行快
- 集成测试验证模块协作
- 端到端测试覆盖关键业务链路

## 2.2 三类测试怎么区分

1. **单元测试（Unit Test）**
   - 测单个函数或类
   - 不依赖真实网络/数据库（可 mock）

2. **集成测试（Integration Test）**
   - 测模块交互（API + DB）
   - 常用于 CRUD 接口验证

3. **端到端测试（E2E）**
   - 从入口到存储的完整流程
   - 成本最高，数量最少

## 三、pytest 核心能力（必会）

## 3.1 基础断言

```python
def test_add():
    assert 1 + 1 == 2
```

## 3.2 fixture（高频）

作用：构建可复用测试前置条件。

```python
import pytest

@pytest.fixture
def sample_product():
    return {"name": "Game", "target_price": 99.0}


def test_product_name(sample_product):
    assert sample_product["name"] == "Game"
```

## 3.3 参数化测试

```python
import pytest

@pytest.mark.parametrize("a,b,expected", [
    (1, 2, 3),
    (2, 3, 5),
])
def test_sum(a, b, expected):
    assert a + b == expected
```

## 3.4 异常断言

```python
import pytest


def divide(a, b):
    return a / b


def test_divide_zero():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)
```

## 四、FastAPI 项目测试实战思路

## 4.1 接口测试应覆盖什么

以产品 CRUD 为例，建议覆盖：

1. 创建成功（201）
2. 列表查询（200）
3. 单条查询存在/不存在（200/404）
4. 更新成功/更新不存在（200/404）
5. 删除成功/删除不存在（200/404）
6. 参数校验失败（422）

## 4.2 你可以这样组织测试用例

- `tests/test_products_api.py`
  - `test_create_product_success`
  - `test_get_product_not_found`
  - `test_update_product_partial`
  - `test_delete_product_success`

## 4.3 为什么状态码也要测试

因为状态码是 API 合约的一部分。返回值内容对了但状态码错，也属于接口错误。

## 五、数据库与异步测试要点

## 5.1 数据库测试原则

1. 每个测试独立，不相互污染
2. 使用测试专用数据库或事务回滚
3. 测试数据尽量小而清晰

## 5.2 异步测试

FastAPI + 异步 ORM 常见实践：
- 使用 `pytest-asyncio`
- 异步 fixture 管理 session
- 测试中 `await` 真实异步调用

## 5.3 事务回滚思路

每个测试开始前建事务，结束后回滚，这样不会把脏数据留到下一个测试。

## 六、Mock 与依赖隔离

## 6.1 什么时候需要 mock

- 外部 HTTP 调用（如爬虫抓取页面）
- 第三方服务（短信、邮件、支付）
- 时间、随机数等不稳定因素

## 6.2 为什么要 mock

- 提升测试速度
- 减少网络波动导致的 flaky test
- 聚焦当前单元逻辑

## 6.3 典型可测点（你的项目）

- `SteamScraper._extract_price_number()` 纯函数可做单测
- `SteamScraper._parse_price()` 可用静态 HTML 样本测试
- `scrape()` 的异常分支可通过 mock `_fetch_page` 覆盖

## 七、面试高频问答（可直接背）

### Q1：单测和集成测试区别？

A：
- 单测测最小单元，强调快和隔离。
- 集成测试测模块协作，强调真实链路正确。

### Q2：为什么测试要可重复？

A：
- 可重复才能稳定发现回归问题，否则会出现偶发失败难定位。

### Q3：你会测哪些边界条件？

A：
- 空输入、非法参数、不存在资源、超时异常、并发写入冲突等。

### Q4：覆盖率越高越好吗？

A：
- 覆盖率是参考指标，不是目标本身。
- 关键是核心业务路径和高风险代码是否被覆盖。

### Q5：怎么减少 flaky test？

A：
- 减少对网络/时间的依赖，控制测试数据，隔离环境，必要时使用 mock。

## 八、结合你项目的标准回答模板

> “在 Price Monitor 项目中，我会按分层测试思路做质量保障：爬虫解析函数做单元测试，产品 CRUD 接口做集成测试，关键流程再补少量端到端测试。接口测试会验证状态码和响应结构，数据库测试保证用例隔离，避免相互污染。对于外部抓取依赖，我会使用 mock 或固定 HTML 样本，让测试稳定可重复。我的目标不是追求表面覆盖率，而是覆盖核心链路和高风险分支，避免回归问题。”
