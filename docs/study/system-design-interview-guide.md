## 一、实习面试里的“系统设计”考什么

对实习生而言，面试官通常不会让你设计“百万 QPS 超大系统”，而是看你是否有：

1. **分层思维**：知道 API、业务、数据层各自职责。
2. **扩展思维**：新功能增加时，改动范围是否可控。
3. **稳定性意识**：是否考虑超时、重试、缓存、异步。
4. **取舍能力**：知道什么时候该简单，什么时候该引入中间件。

## 二、你项目可讲的整体架构

你可以先画一个简化架构：

```
Client
  -> FastAPI Router
    -> Service（可逐步补齐）
      -> SQLModel/SQLAlchemy
        -> PostgreSQL

Background / Scheduled Jobs
  -> Scraper Module (SteamScraper)
  -> Redis (cache / task state)
```

### 架构亮点（面试可讲）

- Web 接口层和数据访问层已分离
- 爬虫模块独立，采用统一接口可扩展
- Docker Compose 管理多个服务，具备部署基础

## 三、分层架构：为什么这样拆

## 3.1 各层职责

1. **API 层（Router）**
   - 参数校验、状态码控制、返回模型
   - 不写复杂业务逻辑

2. **业务层（Service）**
   - 业务规则编排（如“抓取价格后更新历史 + 告警判断”）
   - 统一事务边界与重试策略

3. **数据层（Repository/ORM）**
   - 对数据库读写
   - 屏蔽 SQL 细节

4. **基础设施层**
   - 数据库、缓存、消息队列、日志监控

## 3.2 为什么要分层？

- **解耦**：改数据库实现时，不影响 API 入参定义
- **可测**：业务层可单测，不必每次起全量服务
- **可维护**：职责边界清晰，协作冲突更少

## 四、缓存设计（Redis）

## 4.1 为什么引入缓存

典型动机：
- 热点数据重复查询多
- 读多写少
- 接口对响应延迟敏感

在价格监控场景可缓存：
- 商品详情（短 TTL）
- 最近价格快照
- 热门商品列表

## 4.2 常见缓存策略

1. **Cache Aside（旁路缓存）**（最常用）
   - 读：先查缓存，miss 再查 DB 并回填
   - 写：先写 DB，再删缓存

2. **Write Through**
   - 写请求同时落缓存和 DB（一致性更好，复杂度更高）

3. **Write Back**
   - 先写缓存，异步刷库（高性能但有数据丢失风险）

## 4.3 高频追问：缓存三大问题

- **缓存穿透**：查不存在数据导致请求打到 DB
  - 方案：布隆过滤器、空值缓存
- **缓存击穿**：热点 Key 过期瞬间大量并发打 DB
  - 方案：互斥锁、逻辑过期
- **缓存雪崩**：大量 Key 同时过期
  - 方案：过期时间加随机抖动、限流降级

## 五、消息队列设计（异步解耦）

## 5.1 为什么需要消息队列

在你的业务里，“价格抓取”与“通知发送”可以解耦：

- 主流程只负责写库，快速返回
- 通知、报表、审计日志等走异步消费

## 5.2 典型链路

```
Price Changed Event
  -> Message Queue
    -> Notification Worker
    -> Analytics Worker
```

## 5.3 面试可讲关键词

- 削峰填谷
- 异步处理
- 失败重试
- 死信队列（DLQ）
- 最终一致性

## 六、核心非功能指标：稳定性与可扩展性

## 6.1 稳定性

你可以讲这 4 点：

1. 超时控制（DB / HTTP）
2. 错误分级与告警
3. 限流与熔断（避免雪崩）
4. 健康检查与可观测性（日志、指标、追踪）

## 6.2 可扩展性

1. 水平扩展：API 服务可多实例
2. 模块扩展：新增平台爬虫只加实现类
3. 数据扩展：表索引、分库分表（了解）

## 6.3 一致性 vs 可用性（实习层面）

可简要说：
- 核心交易场景偏一致性
- 查询类场景可接受短暂最终一致

## 七、面试高频问答（可直接背）

### Q1：为什么后端要分层？

A：
- 降低耦合、提升可维护性、便于测试和团队协作。

### Q2：什么时候适合上 Redis？

A：
- 读多写少、热点明显、对响应延迟敏感时。

### Q3：为什么要用消息队列？

A：
- 异步解耦、削峰、提高主链路吞吐和稳定性。

### Q4：如果价格抓取变慢怎么办？

A：
- 抓取任务异步化、并发控制、失败重试、缓存热点数据、降级返回旧值。

### Q5：实习项目要不要上很复杂架构？

A：
- 不建议一开始过度设计。先保证分层清晰，按瓶颈逐步引入缓存和队列。

## 八、结合你项目的标准回答模板

> “我的系统设计思路是先保证分层清晰：Router 负责协议和校验，业务逻辑尽量下沉到服务层，数据访问通过 ORM 管理。价格监控场景里，查询热点可用 Redis 缓存，价格变动通知适合通过消息队列异步处理，避免拖慢主请求。对于可扩展性，我的爬虫模块是可插拔设计，新增平台只需新增 Scraper 实现并注册映射。整体上先用简单架构跑通，再按性能瓶颈逐步演进。”

## 九、面试前速记清单

1. 讲清分层职责（API / Service / Data）
2. 讲清缓存何时用、怎么失效
3. 讲清消息队列的解耦价值
4. 准备一个“性能瓶颈 -> 架构改进”案例
5. 强调“先简单后演进”的工程取舍
